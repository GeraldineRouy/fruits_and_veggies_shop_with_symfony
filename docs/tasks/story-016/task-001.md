# Tâche #001 - Story #016 : Correction de la génération d'ID des OrderLine

## Objectif

Corriger la cause racine du bug de paiement : la séquence PostgreSQL `order_line_id_seq` est désynchronisée après insertion de données de seed avec des IDs explicites, provoquant des conflits de clé primaire (`UniqueConstraintViolationException`) lors de la création de nouvelles commandes.

## Contexte

- Story #016 : `docs/stories/story-016.md`
- Dépend de : Story #013 (paiement simulé, `OrderController`, templates)
- Nécessaire pour : Tâche #002 (gestion d'erreurs), Tâche #003 (tests)

## Prompt

### Description fonctionnelle

**Problème identifié :**
La migration `Version20260524210000` insère des `order_line` avec des IDs explicites (10001-10009) pour les données de seed. En PostgreSQL, `GENERATED BY DEFAULT AS IDENTITY` n'avance PAS la séquence implicite `order_line_id_seq` lors d'INSERTs avec ID explicite. La séquence reste à sa valeur initiale (1). Lorsque l'application crée une commande sans spécifier d'ID, la séquence génère des IDs qui peuvent entrer en conflit avec des enregistrements existants.

**Solution :**
Créer une migration Doctrine qui réinitialise la séquence `order_line_id_seq` à `MAX(id) + 1` de la table `order_line`.

**Cas nominaux :**
- La migration s'exécute et resynchronise `order_line_id_seq` avec le max ID existant
- La migration peut être rejouée sans erreur (idempotente)
- Après migration, les nouvelles OrderLine créées via `OrderService::createFromCart()` reçoivent un ID unique sans conflit

**Cas limites :**
- Si la table `order_line` est vide, la séquence est remise à 1
- Si la migration a déjà été appliquée, elle ne fait rien (idempotence)

**Gestion d'erreurs :**
- La migration doit utiliser `$this->addSql()` avec une requête SQL native PostgreSQL
- Utiliser `set::start()` pour le restart: `SELECT setval('order_line_id_seq', COALESCE((SELECT MAX(id) FROM order_line), 1))`

### Fichiers concernés

| Fichier | Action | Description |
|---------|--------|-------------|
| `migrations/Version20260525120000.php` | Créer | Nouvelle migration pour corriger la séquence |
| `src/Entity/OrderLine.php` | Vérifier | L'entité a déjà `GeneratedValue(strategy: 'IDENTITY')` — ne pas modifier |

### Contraintes techniques

- **Framework** : Doctrine Migrations (via `bin/console make:migration`)
- **Base de données** : PostgreSQL 16, `GENERATED BY DEFAULT AS IDENTITY`
- **Nom de la séquence** : En PostgreSQL, pour une colonne `id INT GENERATED BY DEFAULT AS IDENTITY`, la séquence implicite est nommée `order_line_id_seq` (convention `tablename_columnname_seq`)
- **Idempotence** : La migration doit utiliser `setval()` qui est idempotent — l'exécuter plusieurs fois est sans danger
- **Ne pas régénérer** : Ne PAS utiliser `bin/console make:migration` pour cette correction — écrire le SQL manuellement, car `make:migration` ne détectera pas un problème de séquence

### Structure de la migration

```php
<?php

declare(strict_types=1);

namespace DoctrineMigrations;

use Doctrine\DBAL\Schema\Schema;
use Doctrine\Migrations\AbstractMigration;

final class Version20260525120000 extends AbstractMigration
{
    public function getDescription(): string
    {
        return 'Corrige la séquence order_line_id_seq après insertions avec IDs explicites';
    }

    public function up(Schema $schema): void
    {
        // Remet la séquence au max(id)+1 pour éviter les conflits de clé primaire
        $this->addSql("SELECT setval('order_line_id_seq', COALESCE((SELECT MAX(id) FROM order_line), 1))");
    }

    public function down(Schema $schema): void
    {
        // Pas de rollback nécessaire — setval est une opération ponctuelle
    }

    public function isTransactional(): bool
    {
        return false; // setval() ne doit pas être dans une transaction explicite
    }
}
```
